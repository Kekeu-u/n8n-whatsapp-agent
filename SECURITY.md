# Security

This workflow touches WhatsApp sessions, webhooks, AI providers, Redis and Supabase. Treat every credential as production-sensitive.

## Supported version

Security fixes target the latest public version of the workflow on `main`.

## Before publishing changes

Run a local secret scan before opening a pull request:

```bash
rg "sk-|AIza|api.?key.*=.*['\"][a-zA-Z0-9]" --type-not md
```

Also check manually for:

- n8n credential IDs and exported `credentials` blocks
- webhook IDs generated by n8n
- WAHA API keys
- Supabase service role keys
- Redis passwords
- OpenAI, Perplexity or calendar tokens
- private URLs, phone numbers or screenshots with real chats

## Reporting a vulnerability

Do not post real secrets or exploit details in a public issue.

Open a minimal issue saying there is a security concern, then coordinate privately with the maintainer.

## Recommended production posture

- Put n8n and WAHA behind HTTPS.
- Keep WAHA protected with an API key.
- Avoid exposing Redis publicly.
- Rotate n8n encryption keys and credentials if a workflow export leaks.
- Use Supabase RLS and least-privilege keys.
- Regenerate webhooks if a public export included old webhook IDs.
- Prefer environment variables or the n8n credential store over hardcoded values.
